项目开发更新流程

0. chanquan所需更新项目

以下以chanquan.data为例

1. 开发流程

开发人员本地开发环境在dev分支修改代码进行初步验证

1.1 开发切换本地代码到dev分支并获取该分支的最新代码

1.2 修改代码并进行验证

在本地开发环境修改代码并进行验证

1.3 验证通过，提交到dev服务器进行验证

(注意：当前的分支为dev)

在执行git push前要着重检查刚才操作的输出里是否提示有冲突

  • 没有冲突，可以直接push
  • 有冲突，开发人员自行解决冲突后，再次执行上述add commit push

1.4 开发人员在开发环境自测

待dev服务器代码更新后开发人员自己进行测试

2. 提交测试流程

开发人员在dev环境验证通过后，内容需要交由测试人员测试时，可择机(在测试人员暂时不使用测试环境的间隙)将代码merge到stage

2.1 确认当前所在分支为dev并获取该分支的最新代码

2.2 切换当前分支到stage

2.3 merge代码到stage分支

2.4 merge后着重观察是否有冲突

着重查看merge后的输出消息里是否有conflict字样

在执行git push前要着重检查是否有冲突

  • 没有冲突，可以直接push
  • 有冲突，开发人员自行解决冲突后，再次执行上述add commit push

2.5 确认没有冲突，将代码push到stage分支

2.6 如果有冲突

手动解决完冲突并提交之后，记得反向merge回dev分支，防止刚才解的冲突在下次合并再次冲突

2.6 测试人员进行测试

stage环境更新完成后，通知测试人员进行测试

3. 更新生产环境

在stage环境测试通过后，可以更新生产环境

3.1 确认当前所在分支为stage并获取该分支的最新代码

3.2 切换当前分支到prod

3.3 merge代码到prod分支

3.4 merge后着重观察是否有冲突

着重查看merge后的输出消息里有conflict字样

在执行git push前要着重检查是否有冲突

  • 没有冲突，可以直接push
  • 有冲突，找相关开发人员确认，解决冲突，再次执行上述add commit push

3.5 确认没有冲突，将代码push到prod分支

3.6 如果有冲突

手动解决完冲突并提交后，记得反向merge回stage分支，防止刚才解的冲突在下次merge再次冲突

3.7 在服务器上下拉远程仓库prod分支代码

生产环境服务器所有项目都绑定在prod分支
进入到项目所在目录
执行git pull命令
等待服务自动重启

3.7.1 更新相关详细说明

  • 各个服务器上都有tortoise 进程在每时每刻检测服务状态，发现服务down掉，会自动进行重启。
    通过执行tortoise tail可以查看当前tortoise的日志。
  • 各个服务器都有一套*-svc脚本，用来进行服务的start stop restart
  • aliyun的chanquan-master服务器在执行git push后tortoise会自动进行更新
  • aliyun的chanquan-slaver服务器需要依次进入到项目所在目录执行git pull
  • tianjian的产权服务器同样需要依次进入到项目所在目录执行git pull
  • chanquan.admin因为项目编译时间比较长，可能会出现ssh连接超时卡死的情况。
    解决方案是执行完git pull后，按Ctrl+C中止该次编译，删除掉项目目录下pid.lock(标记当前是否有进程在启动该服务)文件，这样tortoise检测到服务停止了并且没有服务在启动它，然后会自动把它起来。
